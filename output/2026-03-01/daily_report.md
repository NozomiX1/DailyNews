### 1. [Anthropic/Stanford] Token-level Data Filtering: 7000倍重学成本与预训练安全新范式
**来源**: 机器之心 | **时间**: 2026-03-01 11:30
**价值**: 🌟🌟🌟🌟🌟 **标签**: [发布] [AI安全] [预训练] [研究]
**链接**: https://mp.weixin.qq.com/s/Upp5kdC4Onzr7dqyRz-EDA

> 🎯 **一句话摘要**：挑战传统 RLHF 安全范式，提出在预训练阶段通过 Token 级过滤切除危险知识，证明该方法随模型规模扩大效果越显著（1.8B 模型重学成本增加 7000 倍），且优于事后的机器遗忘技术。

#### 🔹 核心技术/实现逻辑

该研究提出了 **Shaping capabilities with token-level data filtering**，主张在预训练阶段而非事后干预阶段消除模型危险能力。

- **代理任务设置**：以“医学知识”作为危险知识代理，以“生物学知识”作为有益知识代理。两者概念高度重叠，验证精准切除难度。

- **Token 级过滤策略**：区别于传统的文档级过滤，针对特定 Token 序列进行精准打击。
    - **Loss Masking**：模型前向传播时可见危险 Token，但在计算 Loss 和梯度更新时将其掩码忽略。保留了上下文连贯性，但切断了权重更新路径。
    - **Removal**：更激进的策略，直接将危险 Token 替换为特殊的 `<hidden>` 标记。模型既无法学习该知识，也无法在上下文中看到该词汇。

- **弱监督工程实现**：解决海量数据标注难题。
    1.  **特征提取**：利用 Gemma Scope 的 **Sparse Autoencoders (SAE)** 提取模型激活的潜在特征。
    2.  **自动标注**：使用 Claude Sonnet 4 等大模型解释 SAE 特征，识别出与目标领域（如医学）相关的特征，生成少量 Ground-truth 标签。
    3.  **分类器训练**：训练一个双向语言模型作为分类器，对预训练数据进行 Token 级别过滤。实验证实，即使分类器含有噪声或仅基于小模型训练，通过“弱到强泛化”仍能在大规模模型上生效。

#### 📊 实验数据/关键结论

- **Loss-Matched Compute Slowdown (算力阻滞指标)**：衡量攻击者恢复被删除能力所需的额外预训练算力。
    - **Scaling Law**：Token 过滤的效果随模型规模增加而增强。
    - **1.8B 参数模型**：目标领域（医学）的学习效率下降 **7000 倍**。

- **对比机器遗忘 (RMU)**：
    - 基线算法 RMU 在少量对抗性微调下防御迅速瓦解。
    - Token 过滤模型表现出极强韧性，18亿参数模型下，攻击者恢复能力所需的数据量是 RMU 模型的 **13 倍以上**。

- **拒绝训练**：
    - Token 移除策略的模型在学习“拒绝回答医学问题”时，正确率比未过滤基线高 **2 倍**。
    - 打破了“必须先知恶才能拒恶”的传统认知，证明了基于“分布内/分布外”的判断比语义判断更稳健。

#### 💡 独家洞察/局限性

- **物理层面的安全**：从未学过与学过再忘存在本质区别。Token 过滤是在物理算力层面构建壁垒，而非依赖模型的对齐“良心”，这对于开放权重的模型尤为重要。
- **Defense-in-depth（纵深防御）**：作者并不认为该技术可替代 RLHF，而是将其作为安全地基，与后续对齐训练结合。
- **工程启示**：不需要完美的分类器即可实现有效的安全过滤，大幅降低了该方案的落地门槛。

### 2. [DeepMind] Aletheia 智能体：零人工干预攻克FirstProof博士级数学难题
**来源**: 新智元 | **时间**: 2026-03-01 09:00
**价值**: 🌟🌟🌟🌟 **标签**: [AI4Math] [DeepMind] [Agent] [学术研究]
**链接**: https://mp.weixin.qq.com/s/fmISRBN5mViWkgzDc9PxOg

> 🎯 **一句话摘要**：Google DeepMind的Aletheia智能体基于Generator-Verifier架构，在零人工干预下独立解决了6道博士级未解数学难题，标志着AI已具备PhD级别的数学科研能力。

#### 🔹 核心技术/实现逻辑

- **Generator-Verifier 双系统架构**：
  - **Generator（生成者）**：负责大开脑洞，疯狂猜想解题路径。
  - **Verifier（验证者）**：作为冷血的“过滤器”，负责检查生成者的输出。只有通过验证的步骤才会被保留，以此极大降低大模型的“幻觉”问题。

- **自我过滤 策略**：
  - 核心原则是“宁可不解，不可乱解”。当系统无法确信解法的正确性时，会直接输出“No solution found”，而不是强行胡编乱造。这种机制保证了在科研级高精度要求下的可靠性。

- **暴力推演与算力投入**：
  - 基于 Gemini 3 DeepThink 模型（文中所述），采用了极端的暴力搜索策略。智能体会在成千上万条死胡同中尝试，直到找到正确路径或耗尽算力。例如，解决P7问题时消耗的算力是此前解决Erdős-1051问题的 **16倍**。

- **零人工干预**：
  - 整个解题过程完全自动化，人类专家未参与任何中间步骤的引导，仅在最后阶段进行验证。

#### 📊 实验数据/关键结论

- **FirstProof 挑战赛成绩**：在10道世界级未解难题中，Aletheia独立解出 **6道**（P2, P5, P7, P8, P9, P10）。相比之下，OpenAI最强内部系统在有限人类监督下解出5道，而其他基准模型（如Gemini 3.0 DeepThink, ChatGPT 5.2 Pro）仅解出2道。

- **具体案例 P7（代数拓扑/微分几何）**：
  - **问题**：判断包含二阶扭转元素的半单李群均匀格，能否作为万有覆盖在有理同调下无圈的紧致无边界流形的基本群。
  - **AI解法**：提供了两种证明路径（拓扑方法与几何方法），利用Lefschetz数矛盾和对称空间刚性定理，被数学家评价为“完美无瑕地串联运用了极其深奥的定理”。

- **推理成本**：解题过程极其消耗算力，绝大多数题目的推理成本均远超此前的极值（Erdős-1051），展示了AI在长链路思维上的投入。

#### 💡 独家洞察/局限性

- **科研范式的转移**：陶哲轩指出，AI正在将数学研究从“个案研究”转向“大样本普查”。AI不知疲倦地处理繁琐计算和扫描问题长尾（如埃尔德什问题集），这是人类无法做到的。

- **Verifier 的决定性作用**：本次成功的关键不单纯在于模型的生成能力，而在于验证器的可靠性。这为未来Agent系统的开发提供了重要思路：在高精度领域，对输出的自我审查能力比单纯的创意生成更重要。

- **局限性**：尽管解决了6道题，仍有4道题未解出，且算力成本极高。此外，目前的系统擅长在已有理论框架内进行复杂的逻辑推演，但在提出全新的、颠覆性的数学概念方面仍依赖人类。

### 3. [Anthropic] Claude Code: Git Worktrees 并行开发与 Plan Mode 质量把控
**来源**: 新智元 | **时间**: 2026-03-01 10:49
**价值**: 🌟🌟🌟🌟 **标签**: [AI编程] [工程实践] [工作流优化] [Anthropic]
**链接**: https://mp.weixin.qq.com/s/6E0FhZFy7xxeUM4k4v_Imw

> 🎯 **一句话摘要**：通过 Git Worktrees 实现多会话并行开发，利用 Plan Mode 与双 AI 互博机制提升代码质量，将 Claude Code 从单线程聊天工具升级为并行“军队”。

#### 🔹 核心技术/实现逻辑

- **Git Worktrees 并行化架构**：
  摒弃传统的单 Repository 单分支开发模式，利用 `git worktree` 创建 3-5 个共享同一个 `.git` 文件夹的独立工作目录。每个目录绑定一个独立的 Claude Code Session，分别运行不同的任务（如：一个跑耗时测试，一个修 Bug，一个重构，一个写新功能）。这解决了传统 AI 辅助中 50% 时间浪费在“等待响应”或“测试运行”上的痛点，实现了真正的异步并行开发。

- **Plan Mode 与“养蛊式”双机互博**：
  - **Plan Mode (Shift+Tab)**：针对复杂任务，强制先进入计划模式，倾注精力规划架构与步骤，再进行 1-shot 执行，避免 AI 在复杂依赖中“走偏”。
  - **Reviewer 机制**：启动两个独立的 Claude Session。Session A 负责出计划、做实现；Session B 使用更干净的上下文扮演“Staff Engineer Reviewer”，专门检查边界条件、错误处理、风格一致性和可维护性。审查意见反馈给 Session A 修改。这种“AI 审查 AI”的机制能有效产出大师级架构设计。

- **资产化记忆**：
  建立 `CLAUDE.md` 作为项目的“施工手册”，硬性规定将代码风格、目录约定、常用命令、过往踩坑经验沉淀其中。完成开发后，专门运行“技术债清扫”Workflow，让 AI 更新该文件。确保新会话无需重复解释背景，直接继承经验。

- **MCP 协议消除上下文孤岛**：
  利用 MCP (Model Context Protocol) 打通 Slack、CI 日志与 IDE。让 AI 直接读取相关上下文和日志，避免开发者进行人肉搬运，专注于决策而非数据搬运。

#### 📊 实验数据/关键结论

- **等待时间减少**：通过并行化，减少了约 50% 用于等待 AI 响应或代码运行的时间。
- **质量提升**：采用 Plan Mode + Reviewer 机制后，生成代码在边界条件处理和架构设计上显著优于单次输出，质量可“碾压”部分资深架构师。
- **效率瓶颈**：若不使用并行模式，开发者极易陷入“输入-报错-回炉修改-测试”的单线程死循环。

#### 💡 独家洞察/局限性

- **局限性**：“养蛊式”开发（双 AI 互博）若盲目使用，可能会增加“代码熵”（Code Entropy）。一旦发现跑偏，必须立刻暂停，回退到 Plan Mode 重新定义问题，而不是在错误的路径上继续修补。
- **部署建议**：不要把 AI 当作简单的“代码补全工具”，而应将其视为“团队成员”。开发者的核心职责从写代码转变为“任务拆解”、“并行调度”和“质量验收”。

### 4. [Westlake University] Rectified Flow Inversion: PMI (Prox-Mean-Inversion) & Zero-Cost Stability
**来源**: 量子位 | **时间**: 2026-03-01 12:00
**价值**: 🌟🌟🌟🌟 **标签**: [ICLR'26] [生成式模型] [Rectified Flow] [图像编辑] [工程优化]
**链接**: https://mp.weixin.qq.com/s/PLtIBzdY_ehX4CptmvDOhQ

> 🎯 **一句话摘要**：西湖大学提出 PMI (Prox-Mean-Inversion) 算法，通过 Proximal Operator 修正逆向轨迹，无需训练即可解决 Rectified Flow 模型的反演不稳定问题，适配 Flux/SD3 等主流架构。

#### 🔹 核心技术/实现逻辑

*   **Rectified Flow (RF) 的痛点**：RF 模型在正向生成（ODE推理）时表现优异，但逆向过程对误差高度敏感，容易偏离前向轨迹，导致图像重建失败。现有的解决方案通常需要额外训练或修改模型结构，工程成本高。

*   **PMI (Prox-Mean-Inversion) 机制**：
    *   **核心原理**：利用 RF 模型训练过程中固有的“平均流”结构。在逆向 ODE 的每一步积分后，引入一个基于 Proximal Operator 的修正项。
    *   **数学直觉**：通过轻量级投影，将当前 latent 拉回到模型的高密度区域（即平均流估计值附近），从而抑制逆向轨迹的偏移与发散。
    *   **工程特性**：完全解耦于原模型推理流程，不引入新参数，不改变模型结构，时间复杂度无额外增加，即插即用。

*   **Mimic-CFG 策略**：
    *   针对 CFG (Classifier-Free Guidance) 在编辑时易导致身份丢失或伪影的问题，Mimic-CFG 将带引导的速度场投影到历史平均方向上。
    *   实现动态补偿，仅在必要维度响应编辑指令，确保“给猫戴帽子”时，猫的身份和背景保持不变。

#### 📊 实验数据/关键结论

*   **评测基准**：在 PIE-Bench（图像编辑权威基准）上与 Vanilla RF-Inversion、FireFlow、Euler 采样器对比。
*   **重建质量**：在相同采样步数（如 20 步）下，PMI 在 **PSNR**（峰值信噪比）上显著提升，像素级还原更精准；在 **LPIPS**（感知相似度）上取得最低分，视觉一致性更强。
*   **泛化性验证**：成功适配 **Flux.1-schnell/dev** 和 **Stable Diffusion 3** 等基于 Rectified Flow 的主流模型，无需微调即可稳定工作。

#### 💡 独家洞察/局限性

*   **工程价值**：随着 Flux 和 SD3 等 Flow Matching 架构成为主流，Inversion（反演）是图像编辑和视频一致性的基石。PMI 提供了一种真正的“Free Lunch”，对于工业界快速落地可控生成系统极具意义。
*   **局限性**：文章未提及对视频生成模型中长序列反演的具体性能表现，主要集中在静态图像编辑任务上。

- **自进化机制**：

  - 核心代码仅 3000+ 行（核心模块数百行），全代码约 8000 tokens，使得 LLM 能够全量理解自身源码，实现了“代码即文档、即社区”的工程理念。
  - **极致压缩 Prompt**：通过分层索引加载记忆，确保 Context 中不包含重复技能定义等垃圾信息，大幅降低 Token 开销。

  - **现场工具制造**：支持动态安装 Python 包、编写脚本并验证方案以解决现有工具不足的问题。
  - **浏览器接管策略**：通过 JS 插件直接接管用户当前浏览器会话（继承登录状态），而非开启全新的无痕实例，解决了网页自动化中的登录验证痛点。

- **跨平台与模型无关性**：支持 PC 与 Android 手机运行；不绑定特定基座模型，支持将习得技能提炼为记忆进行跨设备移植。

#### 📊 实验数据/关键结论

- **代码效率**：核心代码 3000+ 行 vs 传统 Agent 架构 50 万行。
- **上下文占用**：全代码约 8000 tokens，仅占 200K 上下文的 4%。
- **任务表现**：成功演示自主操作微信发朋友圈、手机点奶茶、复杂网页表格填写等交互任务。

#### 💡 独家洞察/局限性

- **技术点评**：该工作体现了“工程架构 > 参数规模”的潜力，特别是“代码即上下文”的思路减少了 Agent 对外部 RAG 文档库的依赖。其“接管现有浏览器”而非“开启无痕浏览器”的思路极具实战价值，直接绕过了复杂的验证码和登录流程，适合个人助理场景。

### 12. [多家大厂] AI Agent 趋势：从Chatbot向系统级执行能力的集体跃迁
**来源**: 量子位 | **时间**: 2026-03-01 10:00
**价值**: 🌟🌟 **标签**: [快讯] [行业] [AI Agent]
**链接**: https://mp.weixin.qq.com/s/ZX-Z52voJI6jdGNbZx8Tlg

> 🎯 **一句话摘要**：硅谷巨头（Meta/Anthropic/Microsoft/Notion等）集体从对话模型转向具备操作能力的 Agent（“Claw”），标志着 AI 从“知识中心”向“执行中心”的产业级转移。

#### 🔹 核心技术/实现逻辑

过去半个月内，多家厂商发布了具备实际操作能力的 Agent 系统，技术重心从生成内容转向调度工具：

- **Meta (Manus Agent)**：核心聚焦于 **Long-term Memory（长期记忆）**。Agent 接入 Telegram 生态，旨在记住用户的交互风格与偏好，结合 Gmail 和 Notion 完成跨平台任务流（如视频制作全流程）。
- **Anthropic (Claude)**：在 Claude Cowork 基础上，过去 48 小时内密集迭代。推出了**手机远程操控代码**及**自动化定时任务**，利用强化学习（RL）和思维链 提升逻辑一致性以支撑复杂任务。
- **Microsoft (Copilot Tasks)**：基于 Windows 生态的深度整合。具备**自主计划**（根据日程生成本周计划）、**跨应用操作**（Outlook -> Calendar -> PPT）及 **Cron Jobs（定时任务）** 能力，主要依赖系统级权限调用。
- **Notion (Custom Agents)**：强调**全自动化与 7x24 待命**。通过设定触发条件（如 Slack @）自动启动任务。底层通过 **MCP 协议** 实现与 Slack、Figma、HubSpot 等工具的自由穿梭与数据交互。
- **Perplexity (Computer)**：主打 **All-in-One**，将搜索、研究、编码、部署整合在单一流程中，实现从想法到上线的闭环。

#### 📊 实验数据/关键结论

- **Notion Agent 渗透率**：早期用户已创建超过 21,000 个 Custom Agents；Notion 内部有 2,800 个 Agent 处于轮班工作状态。
- **行业逻辑转变**：Scaling Law 边际效应递减，行业重心从“模型参数量/知识量”转向“任务执行成功率/系统权限控制”。

#### 💡 独家洞察/局限性

- **信任阈值跨越**：技术落地的关键在于 RL/CoT 推理提升了逻辑一致性，使得人类敢于授予 AI 系统级权限（Notion 甚至提供审计日志）。
- **商业模式重构**：变现逻辑从“卖 Token”（内容生成）转向“卖工时”（劳动力替代）。Agent 直接解决繁琐工时，比单纯的对话更具商业价值。
- **局限性**：文章主要侧重于产品发布与行业观察，未涉及具体的 Agent 架构（如 ReAct vs Plan-and-Solve）或底层工程实现的深度解析。
